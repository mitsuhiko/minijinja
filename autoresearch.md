# Autoresearch: optimize MiniJinja parse + compile benchmark

## Objective
Improve template front-end performance across MiniJinja benchmark workloads.
The optimization target is **parsing and compilation** (lex/parse/codegen/template registration), not render-time VM execution.

## Metrics
- **Primary**: `parse_ns`, `compile_ns` (track both independently and as `parse_compile_ns = parse_ns + compile_ns`)
- **Secondary (guardrail)**: `render_ns`, `render_all_elements_ns`, `render_string_heavy_ns`, `render_macro_heavy_ns`

## Workloads
Current harness reports parse/compile from one representative template and render from three templates:
1. **all_elements.html** — parse benchmark target (`parse_ns`) and compile benchmark target (`compile_ns`)
2. **string_heavy.html** — render regression guardrail
3. **macro_heavy.html** — render regression guardrail

## How to Run
`./autoresearch.sh` — emits `METRIC name=number` lines.

## Files in Scope
- `minijinja/src/compiler/lexer.rs` — tokenization hot paths.
- `minijinja/src/compiler/parser.rs` — AST construction and parser control flow.
- `minijinja/src/compiler/ast.rs` — AST representation/ownership choices affecting compile-time allocations.
- `minijinja/src/compiler/codegen.rs` — bytecode emission and compile-time instruction generation.
- `minijinja/src/compiler/instructions.rs` / `tokens.rs` — compile-time data layout touched by parser/codegen.
- `minijinja/src/environment.rs` / `template.rs` — template registration path that triggers parse+compile.
- `benchmarks/src/bin/autoresearch_render.rs` — benchmark harness producing parse/compile metrics.
- `autoresearch.sh` — benchmark runner.

## Off Limits
- Public API behavior/semantics and Jinja compatibility.
- Feature-gating behavior.
- Runtime-only VM/output/value micro-optimizations unless needed to fix guardrail regressions.
- Crates outside MiniJinja workspace not related to benchmark harness.

## Constraints
- Keep parser behavior, generated bytecode semantics, and rendered output equivalent.
- Preserve error reporting quality (locations/spans and diagnostic behavior).
- Prefer broad optimizations over one-template special-casing.
- No new external dependencies.
- Build must pass for touched code (`cargo check -p benchmarks --bin autoresearch_render`).

## What's Been Tried
- Recent iterations primarily optimized the render path (VM/output/value fast paths).
- For this cycle, prioritize parse/compile internals: lexer scanning, parser allocation patterns, and codegen emission efficiency.
